# Migrations

Inventory of every SQL migration applied to the Weave Supabase schema. Entries are in apply order ‚Äî the same order they're executed by `supabase db push`.

Two projects exist: **Weave-Dev** (`bxbhjybahfyeqytwpkry`) and **Weave** prod (`wndfikmpifyqkgivmnwv`). All migrations below have been applied to both unless otherwise noted. See [`Claude.md`](Claude.md#supabase-environments) for the promotion workflow.

When bootstrapping a new Supabase project, apply these in order. The pgvector extension must be installed before migration 002 (Supabase dashboard ‚Üí **Database ‚Üí Extensions**).

| # | File | One-liner |
|---|------|-----------|
| 001 | `001_create_weave_events.sql` | Behavioral event log ‚Äî `weave_events` table for session + interaction telemetry. |
| 002 | `002_create_weave_embeddings.sql` | Node content embeddings ‚Äî `weave_embeddings(board_id, node_id, embedding extensions.vector(3072), ...)` for semantic similarity. Note the `extensions.` schema-qualified type; Supabase puts `pgvector` there. |
| 003 | `003_create_weave_profile_snapshots.sql` | Reasoning-layer tables ‚Äî `weave_profile_snapshots` (per-run cluster output) and `weave_profile_cluster_embeddings` (one embedding per cluster per snapshot). Powers the Reflect view. |
| 004 | `004_enable_rls_profile_tables.sql` | RLS on reasoning-layer tables ‚Äî default-deny; service-role-only writes until a client-read policy lands (see 006). |
| 005 | `005_cleanup_target_id_format.sql` | One-shot data cleanup ‚Äî removes "ghost board" events (sessions with no content) and normalises legacy `target_id` format. |
| 006 | `006_allow_anon_read_snapshots.sql` | Anon-read policy on `weave_profile_snapshots` so the browser can render the Reflect view. Writes stay service-role-only. To be replaced by `auth.uid()`-scoped policies in the auth pass. |
| 007 | `007_add_archived_at_to_embeddings.sql` | Soft-delete column on `weave_embeddings` (`archived_at timestamptz null`) + partial index `where archived_at is null`. Node deletion sets the timestamp; queries filter it out. |
| 008 | `008_create_core_schema.sql` | Phase 1 cutover tables ‚Äî `boards`, `nodes`, `edges`, `voice_sessions`. Strict RLS from day one (`auth.uid() = user_id`). Shared `update_updated_at_column()` trigger + `updated_at` columns. |
| 009 | `009_add_user_id_to_legacy_tables.sql` | Backfill + NOT NULL on `weave_embeddings.user_id` and `weave_events.user_id`. Index on `user_id`. Anon-permissive policies still in place; strict RLS flip deferred. |
| 010 | `010_create_storage_bucket.sql` | `weave-media` Storage bucket for images/PDFs/audio. Private (signed-URL access). Path convention `{user_id}/...` enforced by RLS on `storage.objects`. |
| 011 | `011_default_user_id_on_legacy_tables.sql` | Default `user_id` on `weave_embeddings` and `weave_events` to the single existing user's UUID. Bridges the 009 NOT NULL constraint with the pre-auth anon-key writer so inserts don't fail silently. Dropped once all writes are authenticated. |
| 012 | `012_board_replace_all_rpc.sql` | `replace_board_contents(p_board_id, p_nodes, p_edges)` PL/pgSQL function ‚Äî atomic replace-all for a board's nodes and edges in one transaction. Touches `boards.updated_at` so sidebar ordering reflects activity. Called from the client via `client.rpc(...)` instead of separate DELETE/INSERT round trips. |
| 013 | `013_allow_authenticated_read_snapshots.sql` | Mirrors the anon snapshot-read policy for the `authenticated` role so signed-in users on prod can render the Reflect view. Superseded by 014, which scopes reads to `auth.uid() = user_id`. |
| 014 | `014_auth_rls_cutover.sql` | Phase 1 auth lockdown ‚Äî adds `user_id` (default `auth.uid()`, FK‚Üí`auth.users` ON DELETE CASCADE) to `weave_profile_snapshots` and `weave_profile_cluster_embeddings`; flips hardcoded UUID defaults on `boards`/`nodes`/`edges`/`weave_events`/`weave_embeddings` to `auth.uid()`; drops every permissive / anon-scoped policy; replaces them with explicit `TO authenticated` SELECT/INSERT/UPDATE/DELETE policies on all seven tables (28 new policies). Service-role writes via Netlify functions still bypass RLS as before. |
| 015 | `015_patch_node_data_rpc.sql` | `patch_node_data(p_node_id uuid, p_board_id, p_user_id, p_patch jsonb)` ‚Äî atomic shallow jsonb merge into `nodes.data`, used by the Fly media-server to write `media_analysis` without racing the client's debounced full-board save. **Superseded by 016 ‚Äî uuid-based lookup was wrong (server only knows the client string id). Apply 015 then 016.** |
| 016 | `016_patch_node_data_by_client_id.sql` | Drops `patch_node_data(uuid,‚Ä¶)` from 015 and recreates it as `patch_node_data(p_client_id text, p_board_id, p_user_id, p_patch jsonb)`, looking up nodes via `data->>'_clientNodeId'`. The Fly server only ever has the ReactFlow string id (the UUID `nodes.id` is server-minted by `replace_board_contents`), so this is the only signature it can call. |
| 017 | `017_replace_board_contents_upsert_nodes.sql` | Switches the node path of `replace_board_contents` from delete-and-reinsert to a prune+upsert keyed on `data->>'_clientNodeId'`. Preserves `nodes.created_at` across saves (was being reset to `now()` on every board write) and lets node UUIDs stay stable across saves, which incidentally makes the patch-by-client-id RPC from 016 hit the same row each time. Edges still wipe-and-reinsert ‚Äî superseded by 018. |
| 018 | `018_replace_board_contents_upsert_edges.sql` | Mirrors 017 for edges: upsert keyed on `(source_node_id, target_node_id, relationship_label)` (the natural triple ‚Äî edges have no client-supplied id), then prune edges this save didn't touch. Same algorithm now everywhere, and edges' `created_at` is preserved across saves when the same (source, target, label) survives. |
| 019 | `019_replace_board_contents_merge_node_data.sql` | Switches the node UPDATE branch from `data = p_data` (full replace) to `data = data \|\| p_data` (shallow merge). Server-added top-level keys like `media_analysis` (written by the Fly server via `patch_node_data`) survive client saves where the client's in-memory state is stale. Tradeoff: client-side deletes of top-level data keys never propagate; current code only adds/updates, never removes. |
| 020 | `020_replace_board_contents_skip_node_noops.sql` | Adds a comprehensive `IS DISTINCT FROM` guard on the node UPDATE so it's a true no-op when no assigned column would actually change (data merge included). Stops `update_nodes_updated_at` from refreshing `updated_at` on debounced saves where nothing changed ‚Äî "last edited" timestamps stay meaningful. |
| 021 | `021_append_processing_log.sql` | `append_processing_log(p_client_id text, p_board_id, p_user_id, p_entry jsonb)` RPC ‚Äî atomic array-append into `nodes.data->'processing_log'` for the Fly media server's structured logger. Uses `jsonb_build_array` + array concat so concurrent calls don't clobber each other (`patch_node_data`'s top-level merge would). Service-role only; soft-fails (raise notice, no exception) when the client node hasn't been persisted yet, since logging is fire-and-forget. |
| 022 | `022_voice_persistence_schema.sql` | Phase 8 voice persistence ‚Äî drops the Phase 2 placeholder `voice_sessions` from migration 008 and replaces it with `voice_sessions(id, user_id, anchor_edge_id, board_snapshot, started_at, ended_at, end_reason, processing_log, summary)` and a new `voice_utterances(id, session_id, user_id, speaker, text, embedding, utterance_index, started_at, ended_at)` table. `anchor_edge_id` is nullable and `ON DELETE SET NULL` so deleting the anchor edge doesn't take the session transcript with it. Unique B-tree on `(session_id, utterance_index)` enforces no-duplicate-or-out-of-order ordering. Four-policy RLS on both tables matching migration 014's pattern. HNSW vector index deferred to 023. |
| 023 | `023_voice_persistence_hnsw_indexes.sql` | Converts `weave_embeddings.embedding` and `voice_utterances.embedding` from `extensions.vector(3072)` to `extensions.halfvec(3072)`, then builds HNSW indexes on both columns using `halfvec_cosine_ops` (pgvector defaults: m=16, ef_construction=64). Lifts pgvector's 2000-dim cap on standard `vector_cosine_ops` by switching to halfvec, which supports HNSW up to 4000 dims at half the storage cost with negligible precision loss for cosine similarity. Retroactively indexes `weave_embeddings`, which had been doing sequential scans since migration 002. `ALTER COLUMN ... TYPE` takes ACCESS EXCLUSIVE during the rewrite ‚Äî sub-second on dev's ~52 rows; re-check prod row count before applying. |
| 024 | `024_voice_utterances_embedding_nullable.sql` | Drops `NOT NULL` on `voice_utterances.embedding`. Embeddings are generated asynchronously: the row is inserted with `embedding = null` as soon as STT/Claude returns text, then a background Gemini call populates the column via UPDATE. Conversation latency never includes the embedding API. Failed embeddings stay null forever (HNSW skips them, retrieval simply doesn't return the row) and are recoverable by re-embedding from `text`; the failure is logged to `processing_log`. |
| 025 | `025_weave_events_voice_session_id.sql` | Adds nullable `voice_session_id uuid` column to `weave_events` plus a partial index `WHERE voice_session_id IS NOT NULL` for the join-back access pattern. Distinct from `weave_events.session_id`, which holds a per-page-load browser correlation id from `services/eventTracker.ts`; the new column points at `voice_sessions.id` (Postgres uuid) when an event was emitted from a voice context. Intentionally no FK constraint ‚Äî telemetry tables shouldn't lose rows when a referenced voice session is deleted. No backfill: pre-Phase-8 weave_events rows have no corresponding voice_sessions row to point at and stay NULL. Populated client-side via `trackEvent({voiceSessionId: voiceSessionController.getSessionId()})` from `voiceSessionLogger`. |

| 026 | `026_backfill_fixture_snapshot_title.sql` | Backfills snapshot titles for fixture rows. |
| 027 | `027_edges_add_mode_column.sql` | **(Edge uniqueness prep, 1/4.)** Promotes edge `mode` ("weave"/"deeper"/"tensions") from `data->>'mode'` to a first-class nullable `edges.mode` column (CHECK: three modes or NULL), backfilled from the jsonb. Dual-write is preserved ‚Äî `data->>'mode'` is still written and still read by hydration's `connectionFromEdge`; the column exists so the directionless unique index and upsert key (028‚Äì030) are cleaner than jsonb-path expressions. |
| 028 | `028_edges_dedup_reconcile.sql` | **(Edge uniqueness prep, 2/4. DESTRUCTIVE.)** Reconciles pre-existing duplicate edges before the unique index. Defines the SHARED CANONICAL identity `(board_id, coalesce(mode,''), least(src,tgt), greatest(src,tgt))` ‚Äî directionless, mode-aware ‚Äî mirrored byte-for-byte in 029, 030, and `src/utils/connectionIdentity.ts`. First-write-wins (`created_at ASC, id ASC` ‚Äî created_at alone is non-deterministic; dev duplicate groups shared a timestamp). Every deleted row is copied whole-row-as-jsonb into `edges_dedup_backup_028` first; recovery query in the migration header. Idempotent. Dev: 96‚Üí93 rows (3 deleted). |
| 029 | `029_edges_unique_directionless_mode.sql` | **(Edge uniqueness prep, 3/4.)** Unique index `edges_unique_directionless_mode` on the canonical expression `(board_id, coalesce(mode,''), least(src,tgt), greatest(src,tgt))`. Unique index on an EXPRESSION (not a column constraint) because direction is normalized via least/greatest. Edge uniqueness is now enforced by construction, not by model instruction. Requires 028 to have run first. |
| 030 | `030_replace_board_contents_directionless_edges.sql` | **(Edge uniqueness prep, 4/4.)** Rewrites the edge upsert in `replace_board_contents` to key on the directionless, mode-aware identity instead of `(source, target, relationship_label)` (migration 018). Fixes two latent bugs: direction-sensitivity (B‚ÜíA minted a second row for an existing A‚ÜíB) and mode-blind label-keying (two real cross-mode connections sharing a label collapsed into one row). Writes the new `mode` column on insert; keeps `data.mode` for hydration. Nodes path unchanged from 018. |

## Regenerating TypeScript types

```bash
npx supabase gen types typescript --linked > src/types/database.ts
```

Run this after any migration that adds/changes tables, columns, enums, or functions. Commit the regenerated file alongside the migration.
